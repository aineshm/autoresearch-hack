# AutoLab — Phase 0 + Track B (L3 Synthesizer) Design Spec

_2026-05-30. Owner: ainesh. Companion to MDDB `hackathon/autoresearch/l2-l3-design` and `l2-l3-work-split`._

> **Hard MVP line.** Hackathon scope (~hours). Each section is tagged **[MVP]** or **[Later]**.
> Build the **[MVP]** slice end-to-end first — a thin loop that round-trips once — then deepen only if time allows.
> The team architecture doc warns explicitly against building all of it up front. Heed that.

---

## 1. Scope

Two deliverables, both JS (ESM) on Node:

- **Phase 0** — the shared spine both tracks depend on: the **blackboard helper module** + the **JSON schemas/fixtures**. Paired with teammate, built first.
- **Track B (L3 Synthesizer)** — ainesh's half: read evidence → judge research integrity → emit a directive. Teammate owns Track A (L2 executor) separately.

Out of scope here: L1 (upstream, writes `program.md`), Track A internals (teammate).

## 2. Stack & integration

- **Language:** **JavaScript (ESM) on Node 24**, both sides — matches the existing repo (backend is `.js` ESM, no TS toolchain). **zod** for runtime schema validation at boundaries (the type-safety value, enforced where JSON contracts actually bite). `node --test` as the test runner. L2 uses the JS `deepagents` package (LangChain.js harness). _(TS is a clean later upgrade — zod schemas yield `z.infer` types for free.)_
- **L3 entry point:** a standalone CLI / function **`synthesize(runDir) → directive`**. Runnable on fixtures with no L2 present. **[MVP]**
- **deepagents integration:** L3 MVP is framework-free (plain function + one LLM-with-tools call via LangChain.js / SDK). Wrapping L3 as a `deepagents` subagent happens **at the live-loop seam, [Later]** — Track B does not inherit Track A's framework-maturity risk.

## 3. The control loop (context)

```
program.md  (north-star: goal, plan, metric, success criteria — write-once by L1)
  → L2 runs ONE pass (lit-review → optimize → evaluate → train/eval), each agent run traced by Raindrop
  → L2 writes results/pass-N.json  (facts)  +  runs/pass-N.jsonl  (per-run monitor summaries)
  → L3 synthesize(runDir): reads { program.md + results + run logs + history }
  → L3 runs integrity check(s) → emits directives/pass-N.json  (+ progress.md [Later])
  → L2 ingests directive → applies changes → loop, unless COMMIT/ESCALATE
```

L3 decides, L2 executes. The blackboard files are the **only** L2↔L3 interface.

## 4. Phase 0 — shared contracts  **[MVP, paired, BUILD FIRST]**

> These schemas are **PROPOSED — pending teammate ratification.** The one field both sides MUST agree on up front is the join key **`run_id` / `raindrop_trace_id`** linking a result to its Raindrop run. All schema access on the L3 side goes **through the blackboard helper** (parse in exactly one place) so a schema change is a one-file edit.

### 4a. Blackboard helper module (`blackboard.js`)  **[MVP]**
A small JS (ESM) module both tracks import. Pure file IO over a `runDir`:
- `readProgram(runDir) → Program`
- `readResults(runDir, pass) → PassResults`
- `readRuns(runDir, pass) → RunRecord[]`
- `writeDirective(runDir, pass, directive)`
- `appendProgress(runDir, entry)` **[Later]**
- `listPasses(runDir) → number[]`
All reads validate against the schema (zod) and fail fast with a clear error.

### 4b. `program.md` (read shape; written by L1)  **[MVP]**
Front-matter + markdown. L3 reads: `{ goal, task_type, metric, direction (min/max), success_criteria, plan }`.

### 4c. `results/pass-N.json` — Experiment Ledger (L2 writes, L3 reads)  **[MVP]**
```json
{
  "pass": 1,
  "candidates": [
    {
      "id": "candidate-1",
      "hypothesis": "gradient-boosted trees baseline",
      "config": { "model": "xgboost", "lr": 0.1 },
      "metrics": { "train": 0.91, "val": 0.86, "held_out": 0.84 },
      "status": "ok",
      "error": null,
      "artifacts": ["models/c1.pkl"],
      "run_id": "rd-abc123"          // join key → Raindrop run + runs/ record
    }
  ]
}
```

### 4d. `runs/pass-N.jsonl` — monitor store (L2/shipper writes, L3 reads)  **[MVP for read; population may be stubbed]**
One JSON line per agent/experiment run:
```json
{ "run_id": "rd-abc123", "raindrop_trace_id": "rd-abc123", "agent_role": "code-optimizer",
  "experiment_id": "candidate-1", "status": "ok",
  "anomalies": ["retried_3x", "skipped_split"], "summary": "..." }
```
L3 reads this file for agent-health evidence. **No live Raindrop API call in MVP.**

### 4e. `directives/pass-N.json` — Decision/Insight (L3 writes, L2 reads)  **[MVP]**
```json
{
  "pass": 1,
  "verdict": "RETRY",
  "checks": {
    "overfit":     { "ok": false, "evidence": "held-out gap 0.07; widening" },
    "stagnation":  { "ok": true,  "evidence": "first pass" },
    "literature":  { "ok": true,  "evidence": "not checked [Later]" },
    "validity":    { "ok": true,  "evidence": "no leakage signal" }
  },
  "changes": [
    { "target": "candidate-1", "action": "set", "field": "max_depth", "value": 4, "reason": "reduce overfit" }
  ],
  "rationale": "Held-out gap signals mild overfit; tighten capacity before exploring.",
  "next_hypotheses": ["lower max_depth + early stopping"]
}
```

## 5. Track B — L3 Synthesizer  **[ainesh]**

### 5a. Deterministic tool substrate (plain testable functions)
- `computeHeldOutGap(results) → {candidateId, gap}[]`  **[MVP]**
- `detectPlateau(history) → bool`  **[MVP]**
- `checkMetricMatchesGoal(program, results) → bool`  **[Later]**
- `checkLiteratureConsistency(results, litFindings) → finding[]`  **[Later]**
These are pure → unit-tested directly. The LLM consumes their output; it never invents the numbers.

### 5b. Integrity checks → verdict
| Check | Source | MVP? |
|---|---|---|
| Overfitting / honest eval | `computeHeldOutGap` | **[MVP]** |
| Cascading-failure / stagnation | `detectPlateau` + `runs/` anomalies | **[MVP]** (results-based MVP; runs/ evidence [Later]) |
| Literature consistency | lit findings | **[Later]** |
| Sanity / leakage / validity | metric-vs-goal + runs/ anomalies | **[Later]** |

**Verdicts:** `CONTINUE / RETRY / PIVOT / COMMIT / ESCALATE`.
COMMIT when all active checks pass AND `program.md` success criteria met (L3 owns goal-attainment for now).

### 5c. `synthesize(runDir)` flow  **[MVP]**
1. `readProgram`, `readResults(latest pass)`, `readRuns` (if present), prior directives for history.
2. Run active deterministic tools → evidence object.
3. One LLM-with-tools call: given {program, results, evidence, history}, fill the directive (verdict + checks + changes + rationale).
4. Validate directive against schema; `writeDirective`.
5. Trace the whole thing in Raindrop **[Later]** (Workshop, local).

### 5d. Raindrop  **[Later, staged]**
- Trace L3's own reasoning (spans per check + verdict) — the "agent debugging its own research" demo.
- Read `runs/` summaries as stagnation/validity evidence (MVP reads the file; live Workshop-MCP drill-down into a flagged `run_id` is Later).

## 5e. Model allocation (product agents) — match tier to reasoning depth

Principle: allocate by the **reasoning effort each role needs**, then pick the cheapest model that clears that bar. Older models are fine where the justification holds — newer ≠ required. Everything set via env (`*_MODEL`) so it's tunable without code changes. "Tier" below is what matters; the named model is a current default with an older fallback.

| Agent | Reasoning effort needed | Tier / default | Older OK | Justification |
|---|---|---|---|---|
| **L1 Formulator** | Medium-high: scope a fuzzy goal into a spec, one bounded pass | workhorse / `gpt-5.4` | `gpt-4o`, `gpt-4.1` | Judgment to narrow intent, but single-shot + checklist-bounded → no flagship needed |
| **L2 parent (conductor)** | Low: dispatch/budget/retry; LLM only for shallow crash diagnosis | mini / `gpt-5.4-mini` | `gpt-4o-mini` | Mostly mechanical orchestration; the LLM seam is "why did this crash," not deep reasoning |
| **Literature reviewer** | Medium: retrieve + summarize papers | mini / `gpt-5.4-mini` | `gpt-4o-mini` | Depth lives in the tools (arXiv/Scholar); model just summarizes → mid-tier suffices |
| **Code optimizer** | High: edit train.py correctly per directive | workhorse / `gpt-5.4` | `gpt-4o`, `gpt-4.1` | Code edits with correctness stakes; a weak model here seeds cascading failures |
| **Evaluator** | High: adversarial review of the optimizer's diff | workhorse / `gpt-5.4` | `gpt-4.1`, `o4-mini` | The guard before code runs — needs genuine reasoning; o-series justified by the review depth |
| **L3 Synthesizer (ainesh)** | Highest: cross-experiment judgment, verdict, integrity gatekeeping | workhorse default / `gpt-5.4`; escalate to flagship `gpt-5.5` or `o`-series on hard passes | `gpt-4o` for MVP | The strategic brain — the one place top-tier reasoning earns its cost. Deterministic tools do the measuring, so the LLM spends tokens on judgment, not arithmetic |

Notes:
- **MVP L3** runs on `OPENAI_MODEL` (default `gpt-4o-mini` from the existing backend) so it works on current credits today. The deterministic tool substrate means even a cheap model yields **sound evidence** — only verdict *quality* scales with tier, so MVP is correct-but-modest and upgrades by changing one env var.
- **Reasoning (o-series) bill internal tokens at output rates (3–10×)** — justified only for L3/Evaluator on genuinely hard cases; gate behind an explicit "escalate model" flag, never the default.
- Use **Flex/Batch** tiers for non-interactive swarm runs (cheaper, latency-tolerant); reserve **Standard/Priority** for interactive L1 intake and L3 verdicts.

## 6. Testing
- **Tools:** pure unit tests, hand-built inputs. **[MVP]**
- **`synthesize`:** golden fixtures — committed `runDir/` snapshots (program.md + results/ + runs/), assert verdict + that a directive validates. LLM call mocked/recorded for determinism. **[MVP]**
- **Live loop:** real L2→L3→L2 round-trip on one pass. **[Later, paired seam]**

## 7. Collaboration seams (where parallel ends)
1. **Phase 0:** ratify schemas + commit blackboard helper + fixtures (incl. the `run_id` join key). Blocks both.
2. **Mid-build:** any shared-schema change → both update fixtures + a shared changelog.
3. **End:** live loop. Only step needing both halves running.

Rule: touches a shared schema or `blackboard.js` → collaborate; everything else → parallel.

## 8. Build order (MVP-first)
1. **[Phase 0]** `blackboard.js` + zod schemas + 2–3 fixtures each for results/runs/program. (paired)
2. **[Track B]** `computeHeldOutGap`, `detectPlateau` + unit tests.
3. **[Track B]** `synthesize(runDir)` with the overfit + stagnation checks → directive; golden-fixture test.
4. **[Track B]** CLI wrapper `synthesize <runDir>`.
5. **STOP — thin loop works.** Then, only if time: remaining checks → `runs/` evidence → Raindrop tracing → `progress.md` → deepagents subagent wrapper.
